LTE to 2G Downgrade Attack

A Deep Dive into Mobile Network Security

By BRAHMI Sarah Kaouther · April 2026 · 7 min read

TL;DR: We built a complete LTE-to-2G downgrade attack simulator in Python. The attack exploits backward compatibility between LTE and 2G networks. Once downgraded, attackers can capture IMSI, intercept SMS, and decrypt calls using the broken A5/1 cipher. Mitigation: Enable LTE-only mode on your phone.

Introduction: The Hidden Danger in Your Pocket

Every day, billions of mobile devices connect to cellular networks around the world. We trust these connections implicitly, making calls, sending sensitive messages, accessing banking apps, and sharing personal data. But what if I told you that this trust is built on a fragile foundation?

Modern 4G LTE networks are designed with robust security in mind. They use mutual authentication, strong encryption, and sophisticated protocols to protect our communications. However, lurking beneath this secure layer is a decades-old vulnerability that attackers can exploit to strip away all these protections.

This is the story of the LTE to 2G Downgrade Attack, how it works, why it's dangerous, and what we're doing to understand and mitigate it.

The Project: Building a Complete Attack Simulator

As part of our mobile security research, we developed a comprehensive software simulation that demonstrates the entire LTE-to-2G downgrade attack chain. The project was built entirely in Python using Tkinter for the graphical interface and a custom publish/subscribe event bus for inter-module communication.

System Architecture

The project follows a decoupled, event-driven architecture with five core modules:

Module Component Role
events.py Event Bus Central publish/subscribe message broker
ue_simulator.py UESimulator Simulates the victim mobile device state machine
attack_engine.py AttackEngine Orchestrates the attack sequence step by step
dashboard.py Dashboard Dark-themed attacker GUI with live log + attack summary
phone_mockup.py PhoneMockup Realistic phone UI showing victim device status

This architecture ensures clean separation of concerns, the simulation engine never directly imports or references the GUI modules, communicating exclusively through events.

Understanding the Attack

How Does the Downgrade Work?

4G LTE is a secure, encrypted mobile standard. However, it's backward-compatible with legacy 2G/GSM networks. Attackers exploit this by:

  1. Setting up a rogue base station (IMSI catcher) that mimics a legitimate cell tower
  2. Sending a fake RRC Connection Reject message to the victim's device, forcing it to disconnect from LTE
  3. Waiting for the device to automatically scan for available networks
  4. Connecting to the attacker's rogue 2G BTS when the phone searches for networks

The Security Gap

The core vulnerability lies in the authentication difference between network generations:

Feature LTE (4G) 2G (GSM)
Mutual Authentication ✅ Yes ❌ No
Strong Encryption ✅ AES ❌ A5/1 (broken)
IMSI Protection ✅ Encrypted ❌ Plaintext
Downgrade Protection ✅ Yes ❌ No

The result: Phones will connect to any tower that claims to be legitimate, making them vulnerable to rogue base station attacks.

Inside the Attack Chain

Phase 1: Protocol Analysis

The attack begins by analyzing the LTE RRC (Radio Resource Control) protocol, which controls cell selection and network switching. Using software-defined radio (SDR) for signal monitoring, attackers can identify the RRC Connection Reject message, a legitimate mechanism designed to deny connections.

Phase 2: Signal Manipulation

Attackers inject a fake RRC Connection Reject message to the victim's device, forcing it to disconnect from LTE. The phone accepts this fake reject, disconnects, and begins searching for alternative networks.

Phase 3: Rogue Base Station Setup

Using OpenBTS (open-source software that transforms SDR into a functional 2G tower), attackers configure a rogue base station with:

  • Valid MCC/MNC (country + network codes)
  • A network name matching the legitimate operator
  • Stronger signal strength than distant real towers

The phone detects what appears to be a legitimate network, trusts it implicitly, and connects, all without user notification.

Phase 4: Exploitation

Once connected to the rogue 2G tower, attackers can access:

Target Risk
IMSI Capture Track individual's location and movements
SMS Interception Read plaintext messages, intercept 2FA codes
Call Metadata Build social graphs and call patterns
Weak Encryption Decrypt voice calls and data (A5/1 cipher)

The Technical Implementation

The Event Bus System

The backbone of our simulation is a lightweight publish/subscribe messaging system. All modules communicate exclusively through this central bus:

events.py ~
# Event bus implementation
_listeners = {}

def subscribe(event_type, callback):
    if event_type not in _listeners:
        _listeners[event_type] = []
    _listeners[event_type].append(callback)

def publish(event_type, data=None):
    if event_type in _listeners:
        for callback in _listeners[event_type]:
            callback(data)

This decouples the simulation engine from the GUI layer entirely, the AttackEngine never imports Dashboard or PhoneMockup, it simply publishes events, and any subscriber reacts independently.

Thread Safety

The UESimulator runs on a background thread to avoid blocking the GUI event loop. All UI mutations are wrapped in root.after(0, update) calls, which schedule updates on the Tkinter main thread, the standard and correct approach for thread-safe Tkinter updates.

Why This Matters: Real-World Implications

Historical Context

The A5/1 cipher was introduced in the 1980s and has been publicly broken for years. Rainbow table attacks and known-plaintext attacks can recover the 64-bit session key in seconds to minutes. Once the attacker holds the IMSI and the session key, they can:

  • Track the device's location and movements
  • Intercept calls and SMS messages
  • Impersonate the victim on the network
  • Access 2FA codes sent via SMS

Law Enforcement and Malicious Actors

IMSI catchers (Stingrays) are used legally by law enforcement with warrants in many countries. However, malicious actors also exploit these vulnerabilities for unauthorized surveillance, targeted tracking, and data theft, a documented global problem.

The Ongoing Threat

As long as 2G networks exist and phones support fallback, this vulnerability remains active. While 3GPP introduced countermeasures in later LTE releases, the attack surface persists.

Defense and Mitigation

User-Level Protections

Enable LTE-Only Mode: Modern Android/iOS settings allow users to disable 2G fallback entirely. This forces the phone to stay on 4G/5G or lose service, preventing downgrade attacks.

settings ~
📱 Settings → Mobile Network → Preferred Network
✓ Select "LTE/5G Only"
# This prevents the phone from connecting to 2G networks

Infrastructure Solutions

5G Standalone (SA): The 5G SA architecture removes 2G/3G support entirely, eliminating legacy protocols and forced downgrade vulnerabilities. Major carriers are transitioning to 5G SA by 2025-2026.

Network Operator Actions: Carriers can disable 2G networks in regions with sufficient 4G/5G coverage, reducing the attack surface for all users. 2G shutdowns are already ongoing in Europe, the US, and Australia.

Detection Systems

Network anomaly detection can identify rogue base stations. Mobile apps like SnoopSnitch and Cell Spy Catcher alert users to potential IMSI catchers in their area.

Key Takeaways

  • Vulnerability Confirmed: We successfully demonstrated the LTE to 2G downgrade attack in a controlled lab setting.
  • Easy to Execute: Requires only commodity SDR hardware (~$300) and open-source software.
  • Solutions Exist: User-side protections are available today, and infrastructure fixes are in progress.
  • Ongoing Threat: As long as 2G networks exist, this vulnerability remains a genuine security concern.

Conclusion

This project successfully demonstrates the complete LTE-to-2G downgrade attack lifecycle in a purely software-based, educational simulation. The event-driven Python architecture cleanly separates the simulation logic from the GUI layer, making the codebase extensible and well-structured.

The dual-window interface (Attack Dashboard + Phone Mockup) provides two simultaneous perspectives on the same attack: the attacker's operational view and the victim device's user experience. This makes the security implications immediately visible and intuitive.

From a security perspective, this simulation highlights a real and historically exploited vulnerability in mobile network standards. The A5/1 cipher used by 2G GSM networks is considered cryptographically broken, and the lack of mutual authentication in early LTE RRC procedures enabled downgrade attacks until 3GPP introduced countermeasures in later releases.

Understanding these attack vectors is essential for mobile security research, network operator defense, and regulatory policy. As we transition to 5G and beyond, we must remember the lessons learned from legacy protocols and ensure that security is built into the foundation of next-generation networks, not bolted on as an afterthought.
