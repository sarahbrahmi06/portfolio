The Challenge

Why AI-Powered Cybersecurity?

Traditional cybersecurity tools are rule-based — they detect known attack patterns but fail against novel or adaptive threats. Meanwhile, enterprise networks are growing in complexity, making manual security monitoring increasingly impractical.

Our project addresses both challenges by combining Network Digital Twin technology with Adversarial Reinforcement Learning — two AI agents that learn attack and defense strategies autonomously through trial and error, discovering tactics that rule-based systems would miss.

System Design

The Three-Layer Architecture

Our platform consists of three interconnected layers working in harmony:

1. Virtual Enterprise Network

18-node 3-tier network emulated in GNS3 with OSPF routing, 5 VLANs, and a victim VM with SSH + Apache.

GNS3 · Cisco IOS · Alpine Firewall

2. Digital Twin & Monitoring

Real-time metrics collection via SNMP (Telegraf), time-series storage (InfluxDB), and a live Grafana security dashboard.

NetBox · Telegraf · Grafana · Ansible

3. RL Control Plane

Two competing DQN agents — Red (attacker) and Blue (defender) — that observe network state and execute actions via REST API.

PyTorch · DQN · Gymnasium

Manual Penetration Testing

The Pre-Game Warm-Up

Before unleashing our AI agents, we had to prove the network was vulnerable and establish a baseline. We conducted manual penetration testing from our Kali machine, targeting the "Victim-1" VM.

The Hydra SSH brute force attack cracked the victim's weak password in just 16 seconds — this established the baseline that the RL Red Agent would need to beat.

kali@attacker — hydra
$ hydra -l victim -P /usr/share/wordlists/rockyou.txt ssh://10.20.20.200 -t 4 # SSH Brute Force Attack [22][ssh] host: 10.20.20.200 login: victim password: 123456 [STATUS] 1 of 1 target successfully completed, 1 valid password found Hydra finished at 2026-05-28 12:27:52 ⚡ 16 seconds — password cracked on first attempt (#1 in rockyou.txt)

● Key Finding: All five attack types (Nmap, curl, Nikto, Hydra, hping3) succeeded against the undefended victim VM. The full attack path traverses 5 network hops across ZeroTier, the Alpine firewall, Router2, and SwitchL32. This established the baseline that the RL defender must learn to prevent.

The Intelligence

How the RL Agents Work

Both agents use Deep Q-Networks (DQN) with 80-dimensional state observations from the network telemetry. They operate in a self-play loop — Red acts first, then Blue responds, and both learn from every step.

Red Agent — Attacker

18 attack actions across 4 kill chain phases: Reconnaissance, Intrusion, Lateral Movement, and Impact. Each action has Severity (damage) and Stealth (detectability) parameters.

● Port Scan   ● ARP Poisoning   ● DDoS   ● Config Tampering
Reward: +50 for Kill Chain Completion

Blue Agent — Defender

18 defense actions across 3 categories: Immediate Response, Recovery, and Proactive/Intelligent. Each defense counters specific Red attacks — Blue must learn the correct attack-defense mapping.

● Port Isolation   ● Rate Limiting   ● Honeypot Deployment   ● Anomaly Detection
Reward: +30 for Chain Interruption

The Results

What We Achieved

After training, the agents demonstrated the ability to learn complex, multi-step strategies. The Red agent learned to chain reconnaissance actions before launching a major attack, mirroring real-world hacker tactics. The Blue agent learned to recognize early warning signs and respond proactively — not just react to outages.

The Future

Conclusion & Next Steps

We've proven that combining a Network Digital Twin with Adversarial Reinforcement Learning is a viable and powerful approach to cybersecurity R&D. We built a platform where AI can safely learn to break and fix a network, developing strategies that rule-based systems would likely miss.

⚠️ Current Limitations

  • • RL training requires GNS3 to be running
  • • 60s SNMP polling may miss rapid attack bursts
  • • Agents operate on interface-level actions only

🚀 Future Work

  • • Integrate real tools: Nmap, Hydra, hping3
  • • Add IDS alerts (Snort/Suricata) to observation space
  • • Reduce SNMP polling to 10s for better burst detection
  • • Add OPNsense firewall for multi-zone scenarios
