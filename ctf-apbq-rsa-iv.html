Three Envelopes, One Safe

Cracking APBQ-RSA-IV (SEKAI CTF 2026)

By Brahmi Sarah Kaouther · July 2026 · 8 min read

TL;DR: The challenge hands you three numbers that look like nonsense. They're not nonsense, they're a single small vector wearing a disguise three sizes too big. Once you notice the disguise, you split that vector into a p half and a q half using a CRT idempotent, chase the split down through one Coppersmith lattice, and finish with a two-line gcd. Total run time: one coffee, one 252-dimensional lattice, and a moment of genuine delight when six equations that should disagree all quietly agree with each other instead.

The Setup

The challenge source, boiled down to the part that matters:

challenge.py ~
p, q = getPrime(1024), getPrime(1024)
n = p * q
e = 0x10001
hints = []
for _ in range(3):
    a, b = randint(0, 4**312), randint(0, 4**312)
    hints.append(a * p + b * q)
c = pow(bytes_to_long(FLAG), e, n)

Three public numbers, each one a random blend of p and q with coefficients up to 2624. That's it. That's the whole safe.

First Instinct

Before doing anything fancy, always check whether the challenge author accidentally left a factor lying around. gcd each hint against n. Here: all three come back 1. No shortcuts today. Time to actually work.

Numbers on the table, for reference: n is 2047 bits, the three hints sit at 1647-1649 bits. Nothing screams "small" yet.

Reading the Room: It's One Vector, Not Three Numbers

Stack the three hints into a vector H = (h₀, h₁, h₂), and likewise A = (a₀, a₁, a₂), B = (b₀, b₁, b₂), so H = pA + qB. Now here's the move that unlocks everything: build the lattice

Λ = ⟨H, ne₀, ne₁, ne₂⟩ ⊂ ℤ³

Why the extra neᵢ generators? Because we're allowed to add any multiple of n to any coordinate without breaking anything mod n so those vectors are free filler that don't cost us anything but keep the lattice well-defined over ℤ rather than just mod n.

Since h₀ is invertible mod n, the subgroup that H generates inside (ℤ/nℤ)³ has exactly n elements. The whole group has n³ elements, so Λ's index is n², meaning det(Λ) = n². Small detail, but it's the anchor for every bound we compute later.

Compute a Hermite normal form basis, LLL-reduce it, and call the result V. Because H ∈ Λ by construction, there's an integer vector s with H = sV.

Measured:

  • Coordinates of s: roughly 284, 285, 282 bits
  • Coordinates of the hints: 1647, 1649, 1648 bits
  • Difference: about 1360 bits

Same object, radically smaller description once you're standing in the right basis. This is the whole ballgame in miniature.

Two Envelopes Inside the One Envelope

Here's the part that actually made this problem interesting instead of just mechanical.

pA and qB are each individually invisible to us, we only ever get their sum H. But both of them are secretly members of Λ too, which is not obvious and is the whole reason this challenge isn't trivial. The argument uses a CRT idempotent: an integer εₚ satisfying

εₚ ≡ 0 (mod p),    εₚ ≡ 1 (mod q)

Multiply H by it: mod p, εₚH ≡ 0; mod q, εₚH ≡ H ≡ pA (since qB ≡ 0 there). By CRT, that means εₚH ≡ pA (mod n) so pA sits inside Λ up to an integer multiple of n, which we already folded into the lattice. Same logic hands us qB ∈ Λ.

So there really are integer vectors u, v with uV = pA, vV = qB, and because pA + qB = H = sV we get the clean little identity:

u + v = s

s is a known, public, computed-five-minutes-ago vector. u and v are locked in a vault. But they're not just any two numbers that sum to s they're two very specific lattice vectors, one of which is entirely built from p. The moment we recover u, any nonzero coordinate of uV is a multiple of p, and one gcd call against n ends the challenge.

Writing an Equation That Only the Hidden Vector Can Answer

We need a second constraint on u beyond "it's part of a pair summing to s." Take the first coordinate of the hint (invertible mod n, we already checked) and define a public linear functional:

R(x) = (xV)₀ / h₀ (mod n)

Plug in the hidden u:

R(u) = (p a₀) / (p a₀ + q b₀) (mod n)

Reduce mod p: top line is zero. Reduce mod q: bottom line collapses to p a₀, leaving exactly 1. So R(u) is itself a CRT idempotent, it's zero in one world and one in the other, which means:

R(u)² - R(u) ≡ 0 (mod n)

Two roots of this equation are already known and boring: u = 0 and u = s. The two we actually want are hiding elsewhere. Rather than chase u directly, center the variable: d = 2u - s. Using R(s) = 1 and the identity (2e - 1)² = 1 for any idempotent e, this turns into:

R(d)² - 1 ≡ 0 (mod n)

with four roots total: ±s (known) and ±d (the prize). Nice side effect of centering: the polynomial only involves even-degree monomials from here on, which quietly halves the search space later.

Sizing the Vault Before You Try to Crack It

Since dV = pA - qB, define yᵢ = paᵢ - qbᵢ = 2paᵢ - hᵢ. Given 21023 ≤ p ≤ n / 21023 and 0 ≤ aᵢ,bᵢ ≤ 2624, this pins down an interval for each yᵢ, and from there a bound on d = yV⁻¹.

That raw bound comes out lopsided across the three coordinates, so we apply a small unimodular change of variables z = dP for a hand-picked integer matrix P (determinant 1). Post-transform, all three coordinates land around 282-284 bits.

Not Cheating, Just Tidying: P isn't a secret and isn't derived from anything hidden, it's purely a convenience rotation computed from public bound data, chosen to make the lattice construction behave better.

Building the 252-Row Machine

Substitute d = zP⁻¹ into R, giving L(z) = R(zP⁻¹), and the small-root target becomes:

g(z) = L(z)² - 1 ≡ 0 (mod n)

normalized so the z₀² coefficient is exactly 1. Pick multiplicity m = 4, and take every even-total-degree monomial in (z₀,z₁,z₂) up through degree 12. Counting them:

Σⱼ₌₀⁶ C(2j+2, 2) = 1 + 6 + 15 + 28 + 45 + 66 + 91 = 252

For each monomial zᵉ, insert the shift:

Gₑ(z) = n4-k g(z)k z₀e₀-2k z₁e₁ z₂e₂,   k = min(4, ⌊e₀/2⌋)

Why 252 and not, say, 40?

More rows = more constraints = tighter provable bound on how small a root we can find. Fewer rows = faster reduction but a bound too loose to catch the actual root. 252 is not a magic number, it's just where "the bound covers our actual root size" and "the matrix still finishes reducing before the CTF ends" cross paths.

Panning for Exact Equations

Not every short row out of the reduction is meaningful, some are just short by coincidence. The filter is a Howgrave-Graham style norm check: if a row's coefficient vector, rescaled by the root bounds, has norm (times √# nonzero terms) below n⁴, then the corresponding polynomial doesn't just vanish mod n⁴ at the root, it vanishes exactly, over the integers.

Run every row through that filter and 250 of the 252 survive as genuine, exact, integer-true relations.

The Shortcut: Everything Here Is Rank One

Stack the 250 exact relations into a 250 × 252 matrix C and reduce it mod a disposable 61-bit prime ℓ. Typically rank(C mod ℓ) = 250, leaving a 2-dimensional kernel, spanned exactly by the even-monomial evaluation vectors of the two roots that still matter: the known Eₛ and the hidden Eᵈ.

Pull a basis b₀, b₁ of that kernel, cancel out the constant-monomial coordinate to get a direction w ∝ Eᵈ - Eₛ, and now Eᵈ = Eₛ + tw for a single unknown scalar t.

Here's the trick that makes this tractable: the six quadratic monomials (z₀², z₀z₁, z₀z₂, z₁², z₁z₂, z₂²) are exactly the entries of a rank-one matrix. Rank-one matrices have every 2×2 minor equal to zero , always, no exceptions. That's six free quadratic equations in the single unknown t.

Cashing Out

Undo the coordinate change: d = zP⁻¹ for each of the two possible signs of z. Exactly one sign gives integer coordinates for u = (d + s) / 2; the other gives garbage. For the correct sign, uV = pA, so:

gcd.py ~
for value in uV:
    p = gcd(abs(int(value)), n)
    if 1 < p < n: break

From there it's textbook RSA cleanup:

  • q = n / p
  • φ(n) = (p - 1)(q - 1)
  • d_rsa = e⁻¹ mod φ(n)
  • Decrypt, done.

Both sanity checks passed on the first try:

  • assert p * q == n
  • assert pow(bytes_to_long(flag), 0x10001, n) == c

def parse_instance(path: Path) -> tuple[int, int, list[int]]:
    """Grab the challenge data from the file."""
    text = path.read_text()
    blocks = text.split("'''", 2)
    if len(blocks) < 2:
        raise ValueError("Couldn't find the challenge output block")
    block = blocks[1]
    out = {}
    for name in ("n", "c", "hints"):
        match = re.search(rf"^{name} = (.+)$", block, re.MULTILINE)
        if not match:
            raise ValueError(f"Missing {name} in challenge output")
        out[name] = ast.literal_eval(match.group(1))
    return int(out["n"]), int(out["c"]), list(map(int, out["hints"]))

def public_hint_basis(n: int, H: list[int]) -> tuple[Matrix, tuple[int, int, int]]:
    generators = Matrix([
        H,
        [n, 0, 0],
        [0, n, 0],
        [0, 0, n],
    ])
    basis = hermite_normal_form(generators.T).T
    A = IntegerMatrix.from_matrix([
        [int(basis[i, j]) for j in range(3)] for i in range(3)
    ])
    LLL.reduction(A, delta=0.9999)
    V = Matrix([[int(A[i, j]) for j in range(3)] for i in range(3)])
    s_row = Matrix([H]) * V.inv()
    if any(x.q != 1 for x in s_row):
        raise AssertionError("H isn't integral in the reduced basis")
    s = tuple(int(x) for x in s_row)
    assert abs(int(V.det())) == n * n
    assert Matrix([s]) * V == Matrix([H])
    return V, s

def monomial_value(point, e, mod=None):
    value = 1
    for x, power in zip(point, e):
        if mod is None:
            value *= x**power
        else:
            value = value * pow(x % mod, power, mod) % mod
    return value

Debrief

Four moves, if you strip away the scenery:

  1. The three hints are one vector in disguise find the lattice it actually lives in, reduce, and the disguise falls off.
  2. That short vector is secretly two hidden vectors glued together a CRT idempotent proves it and lets you write u + v = s.
  3. Turn "recover u" into a small-root problem via a second CRT idempotent identity, throw a 252-dimensional Coppersmith lattice at it.
  4. Let the rank-one matrix hand you the missing scalar no full algebraic solve needed.

Trust, but Verify:

  • The bit-length of s matches the predicted compression ratio
  • 250 independent relations agree on a rank-one structure
  • Six separate minors agree on the same scalar t
  • Recovered p, q are checked directly against n and against the ciphertext

Nothing here rides on a single lucky coincidence.

Conclusion

This challenge was a beautiful example of how lattice reduction can be applied to RSA problems with structured hints. The key insights were:

  1. Recognizing that three large numbers could be compressed into one small vector
  2. Using CRT idempotents to isolate p and q components
  3. Building a Coppersmith lattice with the right dimension to catch the hidden root
  4. Exploiting the rank-one structure for the final recovery

The solution demonstrates advanced mathematical and algorithmic thinking, exactly what makes CTFs both challenging and rewarding.
